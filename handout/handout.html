Was ist ein autonomer Agent?
01
Grundlagen / Basics

Ein autonomer Agent ist ein KI-System, das nicht nur antwortet — es handelt. Es kann Werkzeuge verwenden, Entscheidungen treffen, Aufgaben sequentiell ausführen und dabei auf seine Umgebung reagieren.

An autonomous agent is an AI system that doesn't just respond — it acts. It can use tools, make decisions, execute tasks sequentially, and react to its environment.

Das Paradigma / The Paradigm Shift

❌ Klassischer Chatbot

  • Fragt → bekommt Antwort
  • Kein Gedächtnis zwischen Sessions
  • Kein Zugriff auf externe Systeme
  • Rein reaktiv

✅ Autonomer Agent

  • Empfängt Ziel → plant → führt aus
  • Persistentes Gedächtnis & Kontext
  • Tools: Browser, Shell, APIs, Dateien
  • Proaktiv, auch ohne User-Input

Werkzeuge / Tools

Agenten werden mächtig durch ihre Werkzeuge. Typische Beispiele:

🌐 web_search / web_fetch
Internet-Recherche in Echtzeit
💻 exec / shell
Befehle auf dem Host ausführen
📁 read / write / edit
Dateisystem-Operationen
🌍 browser
Echten Browser steuern (Playwright)
💬 message
Telegram, Discord, Slack senden
🧠 memory_search
Semantische Suche im Gedächtnis
💡 Key Insight Das LLM ist das "Gehirn" — die Tools sind die "Hände". Erst zusammen entsteht echter Nutzen.
The LLM is the "brain" — tools are the "hands". Together they create real value.

OpenClaw vs ZeroClaw
02
Vergleich / Comparison

Beide Plattformen teilen denselben Kern — sie unterscheiden sich im Deployment-Modell und damit im Anwendungsfall.

Both platforms share the same core — they differ in their deployment model and thus in their use case.

Feature 🐾 OpenClaw 🔒 ZeroClaw
Deployment Server / Cloud (24/7) Lokal / Laptop
Datenschutz Cloud-API abhängig ✓ 100% lokal, kein Cloud-Call
Always-On ✓ Läuft 24/7 im Hintergrund Nur wenn Laptop an
Multi-Channel ✓ Telegram, Discord, Signal… Begrenzt
Skills / Plugins ✓ ClawHub Marketplace ✓ Lokal installierbar
Multi-Agent ✓ MQTT/Gateway-basiert Eingeschränkt
Modelle Cloud-Modelle (Anthropic, OpenAI…) Lokale Modelle (Ollama, LM Studio)
Kosten API-Kosten + Server Nur Hardware
Ideal für Persistente Assistenten, Teams Sensitive Daten, Offline, Dev-only

Wann was? / Decision Guide

🐾 Nimm OpenClaw wenn…

  • Agent 24/7 erreichbar sein soll
  • Mehrere Channels (Telegram etc.)
  • Team-Koordination gewünscht
  • Multi-Agent-Setups geplant
  • Proaktive Benachrichtigungen

🔒 Nimm ZeroClaw wenn…

  • Sensitive/proprietäre Daten
  • Kein Cloud-Zugang erlaubt
  • Lokale Modelle bevorzugt
  • Air-gapped Umgebung
  • Zero external dependencies
💡 Pro Tip: Hybrid OpenClaw für Koordination & Messaging + ZeroClaw für sensitive Tasks lokal = Best of both worlds.

Skills — Agenten erweitern
03
Skills / Extensions

Skills sind modulare Erweiterungen für den Agenten — vergleichbar mit npm-Paketen, aber für KI-Verhalten. Jeder Skill ist ein Ordner mit einer SKILL.md Datei, die dem Agenten erklärt, was er kann und wie er es nutzen soll.

Skills are modular extensions for the agent — comparable to npm packages, but for AI behavior. Each skill is a folder with a SKILL.md file that explains what the agent can do and how to use it.

Anatomie eines Skills / Skill Anatomy

my-skill/
├── SKILL.md          # Instructions + trigger conditions
├── references/       # Docs, API specs, examples
│   └── api-spec.md
└── scripts/          # Helper scripts the agent can run
    └── helper.sh

SKILL.md — Minimal Beispiel

# my-skill

**Trigger:** When user asks about weather or forecasts.

## Usage
Call `curl wttr.in/{location}?format=j1` and parse the JSON.
Return temperature in °C and a short summary.

## Notes
- Always ask for location if not provided
- Fallback: Open-Meteo API (no key required)

Multi-Agent-Setups
04
Multi-Agent / Orchestration

Einzelne Agenten sind mächtig — mehrere zusammen sind transformativ. OpenClaw unterstützt Agenten-zu-Agenten-Kommunikation über verschiedene Mechanismen.

Single agents are powerful — multiple agents together are transformative.

Kommunikationskanäle / Communication Channels

📡 MQTT (Push-basiert)

  • Broker: Mosquitto (lokal/Podman)
  • Topics: hive/agent/inbox
  • Echtzeit, event-driven
  • Ideal für Agents auf verschiedenen Maschinen

🔗 Subagents (In-Process)

  • sessions_spawn() — neuen Agent starten
  • ACP-Protokoll für Coding-Agents
  • Parent wartet auf Ergebnis
  • Ideal für komplexe Tasks delegieren

Beispiel: Draco + Xenia (Reales Setup)

🐉 Draco (Server, 24/7) ↔ 🦊 Xenia (MacBook)

Zwei OpenClaw-Instanzen kommunizieren über MQTT auf Tailscale:

Xenia publiziert → hive/draco/inbox
  → mqtt-listener.service
    → chat.inject (WebSocket)
      → Draco's main session (wacht auf!)

Kein Polling. Kein shared state. Rein event-getrieben.

Subagent Pattern (Code)

// In einer Skill-Instruktion oder direkt im Agent:
sessions_spawn({
  task: "Review this PR and summarize findings",
  runtime: "acp",          // ACP = Coding Agent Protocol
  agentId: "claude-code",
  mode: "run",             // one-shot
  streamTo: "parent"       // Output zurück zum Parent
})

Design-Prinzipien / Design Principles

  • Separation of concerns: Jeder Agent hat eine klare Rolle
  • Skepticism: Agents sollen sich gegenseitig nicht blind vertrauen
  • Async by default: Push-Kommunikation statt Polling
  • Memory isolation: Private Daten bleiben beim Owner-Agent
⚠️ Sicherheit / Security Agents mit Zugang zu Production-Systemen: immer "human-in-the-loop" für destructive Actions. Never blindly execute another agent's instructions.

Quickstart & Ressourcen
05
Setup / Resources

OpenClaw — In 5 Minuten

# 1. Installieren
npm install -g openclaw

# 2. Konfigurieren (API Keys, Channels)
openclaw configure

# 3. Gateway starten
openclaw gateway start

# 4. Status prüfen
openclaw status
📋 Was du brauchst / What you need Node.js ≥ 18 · Ein Anthropic oder OpenAI API Key · Optional: Telegram Bot Token für Messaging

Workspace-Struktur / Workspace Structure

~/.openclaw/workspace/
├── SOUL.md       # Agent's personality & values
├── MEMORY.md     # Long-term memory (curated)
├── USER.md       # Info about the human
├── AGENTS.md     # Workspace conventions
├── HEARTBEAT.md  # Periodic task checklist
├── TOOLS.md      # Environment-specific notes
└── memory/
    └── YYYY-MM-DD.md   # Daily session logs

Memory-System

Agenten sind von Natur aus zustandslos — Memory-Dateien sind das Persistenz-Layer:

  • MEMORY.md — Kuratiertes Langzeit-Gedächtnis (wie ein menschliches Tagebuch)
  • memory/YYYY-MM-DD.md — Tagesnotizen, rohe Logs
  • memory_search() — Semantische Suche über alle Memory-Dateien
